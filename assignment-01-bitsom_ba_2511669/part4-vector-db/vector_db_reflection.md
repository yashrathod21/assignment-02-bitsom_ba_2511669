## Vector DB Use Case

A traditional keyword-based database search would be fundamentally inadequate for this use case, and here is why.

Keyword search works by matching the exact tokens in a query against an inverted index of the document's words. If a lawyer types "What are the termination clauses?", a keyword search looks for the literal string "termination" across the 500-page contract. This works only when the document uses that exact word. But legal contracts routinely use synonymous and paraphrased language — a termination clause might be titled "Cessation of Agreement", "Exit Provisions", or "Dissolution Conditions" without the word "termination" appearing in the section heading at all. Keyword search would miss all of these entirely. It has no understanding of semantic intent; it matches tokens, not meaning.

A vector database solves this by operating in the space of meaning rather than words. The contract is first chunked into paragraphs or logical sections, and each chunk is converted into a dense vector embedding using a language model such as `all-MiniLM-L6-v2` or a legal-domain fine-tuned model like LegalBERT. These embeddings encode the semantic content of each passage — not just its words. The vectors are then stored and indexed in a vector database such as Pinecone, Weaviate, or pgvector.

When a lawyer submits the query "What are the termination clauses?", that query is also embedded into the same vector space. The vector database performs an approximate nearest-neighbour (ANN) search and retrieves the contract passages whose embeddings are closest in cosine distance — regardless of whether the word "termination" appears. A passage about "exit rights upon material breach" would surface correctly because its meaning is semantically proximate to the query intent.

In this system, the vector database plays the role of the semantic retrieval layer — transforming a plain-English question into a meaningful search across dense, unstructured legal text. This is precisely the task that keyword databases cannot perform reliably, and that vector databases are purpose-built for. The overall architecture — document chunking, embedding generation, vector indexing, and similarity retrieval — is known as Retrieval-Augmented Generation (RAG), and it is the standard approach for building natural-language search over large private document collections.
