## Database Recommendation

For a healthcare startup building a patient management system, I would recommend **MySQL (a relational RDBMS)** as the primary database — with an important caveat for the fraud detection module discussed below.

The core reason is **ACID compliance**. Healthcare data is among the most sensitive and legally regulated data in existence. When a doctor updates a patient's medication record, that update must be atomic — either it fully commits or it fully rolls back. There can be no partial writes where a drug dosage is updated but the allergy flag is not. MySQL's ACID guarantees (Atomicity, Consistency, Isolation, Durability) make this possible by design. MongoDB, while improving its transaction support in recent versions, is fundamentally built around a BASE model (Basically Available, Soft state, Eventually consistent), which trades strict consistency for availability and performance. In a patient management context, eventual consistency is not acceptable — a nurse querying a patient's allergy record must always read the latest committed data, not a stale replica.

The **CAP theorem** reinforces this choice. MySQL in a typical synchronous-replica setup prioritizes **Consistency and Partition Tolerance (CP)**, meaning it will refuse a read or write rather than return stale data. MongoDB by default prioritizes **Availability and Partition Tolerance (AP)**, which is appropriate for product catalogs or social feeds but dangerous for medical records where a read returning old data could result in the wrong medication being administered.

Additionally, patient management systems have well-defined, structured schemas — patient demographics, appointments, diagnoses, prescriptions, lab results — which map naturally to relational tables with foreign key constraints and enforced joins.

**For the fraud detection module, the answer changes significantly.** Fraud detection requires analyzing high-velocity, heterogeneous event streams — login attempts, billing anomalies, unusual access times — in near real-time. This unstructured, rapidly evolving data fits MongoDB's flexible document model far better than a rigid relational schema. A hybrid architecture would be ideal: **MySQL for the core patient management system** (structured, ACID-critical) and **MongoDB combined with a streaming layer like Apache Kafka** for the fraud detection pipeline (high-throughput, schema-flexible). This gives each workload the database model it actually needs without compromising either.
